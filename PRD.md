## 1. Цель и контекст

КТЖ одновременно реализует несколько крупных проектов железнодорожного строительства (новые линии, обходные пути, реконструкция узлов). Данные о ходе работ поступают в виде разрозненных презентаций (PDF) от разных подрядчиков и служб, с разной структурой и периодичностью обновления. У руководства/топ-менеджмента нет единой точки, где можно за 1–2 минуты увидеть:

- текущий % готовности по каждому проекту и по этапам работ;
- риск срыва директивных сроков;
- конкретные причины отставания и принимаемые меры.

**Цель продукта** — дать топ-менеджменту КТЖ единый веб-дашборд для мониторинга статуса строительства крупных проектов с ручным вводом/обновлением данных, без необходимости интеграции с внешними системами на первом этапе.

## 2. Целевой пользователь

| Параметр | Значение |
|---|---|
| Основной пользователь | Руководство КТЖ / топ-менеджмент |

| Технический уровень | Любой — не требует обучения |
| Источник данных | Ручной ввод на основе презентаций/отчётов подрядчиков и служб КТЖ |

## 3. Объём первой версии (Scope v1)

### Входит в v1
- Веб-приложение в виде одного HTML-файла (артефакт), без серверной части
- Два режима отображения: **Карточки** (обзорный) и **Таблица** (для быстрого массового заполнения)
- Полный CRUD по объектам строительства (создание / просмотр / редактирование / удаление)
- Автосохранение данных в персистентное хранилище (привязано к пользователю и артефакту)
- Сводные метрики по портфелю проектов
- Поля для фиксации риска отставания и принимаемых мер по каждому объекту
- Фильтрация по региону, статусу, текстовый поиск

### Не входит в v1 (возможные следующие итерации)
- Автоматическая интеграция с системами подрядчиков / 1С / внутренними БД КТЖ
- Многопользовательский режим с разграничением прав доступа и историей изменений
- Drill-down по участкам/субподрядчикам внутри одного проекта (как в исходных презентациях по Бахты–Аягоз, Мойынты–Кызылжар)
- Графики динамики (Gantt, S-curve по освоению бюджета во времени)
- Экспорт в Excel/PDF, печатные отчётные формы
- Уведомления/алерты при превышении пороков отставания
- Мобильная версия / адаптив для планшетов в полевых условиях

## 4. Пользовательские сценарии (User Stories)

1. **Как руководитель**, я хочу за один экран увидеть, сколько проектов в графике, сколько в зоне риска и сколько отстают — чтобы понять, куда направить внимание.
2. **Как руководитель**, я хочу открыть конкретный проект и увидеть % готовности по этапам, плановую/прогнозную дату завершения и причину отставания — чтобы принять решение.
3. **Как куратор**, я хочу быстро обновить цифры по нескольким проектам сразу в табличном виде — чтобы не открывать каждую карточку отдельно.
4. **Как куратор**, я хочу зафиксировать не только риск, но и какие меры приняты для его снижения — чтобы это было видно руководству без дополнительных созвонов.
5. **Как руководитель**, я хочу отфильтровать проекты по региону или статусу — чтобы сосредоточиться на проблемной географии.

## 5. Функциональные требования

### 5.1 Модель данных проекта

| Поле | Тип | Описание |
|---|---|---|
| `name` | строка | Название проекта/участка |
| `region` | список (фиксированный) | Регион реализации |
| `length` | число | Протяжённость, км |
| `startdate` | дата | Дата начала работ |
| `plandate` | дата | Плановая (директивная) дата завершения |
| `forecastdate` | дата | Прогнозная/фактическая дата завершения |
| `budgetplan` | % (0–100) | Плановое освоение бюджета |
| `budgetfact` | % (0–100) | Фактическое освоение бюджета |
| `stages[5]` | % (0–100) каждый | Готовность по 5 этапам: земляное полотно, искусственные сооружения, укладка ВСП, электрификация и СЦБ, станции |
| `status` | enum | `green` (в графике) / `amber` (риск отставания) / `red` (отставание) / `done` (завершён) |
| `updated` | дата | Дата последнего обновления данных |
| `risk` | текст (многострочный) | Региональная специфика и причины риска отставания |
| `mitigation` | текст (многострочный) | Принимаемые мероприятия по сокращению отставания |

**Производные (расчётные) показатели:**
- `Готовность общая, %` = среднее по 5 значениям `stages`
- `Отставание, дн.` = `forecastdate − plandate` (если обе даты заданы)
- `До срока / просрочено, дн.` = `plandate − текущая дата`

### 5.2 Режим «Карточки»
- Каждый объект — отдельная карточка
- Световой индикатор статуса (зелёный/жёлтый/красный, по аналогии с ж/д сигналом)
- Полоса прогресса по 5 этапам с подписями
- Блок «Риск отставания» (текст с акцентом)
- Блок «Мероприятия по сокращению отставания» (текст с акцентом, отдельным цветом от риска)
- Общая готовность, % и освоение бюджета (план/факт)
- Плановая дата и расчётное «до срока»/«отставание»
- Кнопки «Изменить» / «Удалить»

### 5.3 Режим «Таблица для заполнения»
- Все проекты — построчно, все поля редактируются прямо в ячейке
- Цветовая кодировка: подтверждённые/исходные данные — выделены, расчётные поля — недоступны для редактирования
- Изменение сохраняется автоматически при потере фокуса поля (`onChange`)
- Колонки включают оба новых текстовых поля — риск и мероприятия

### 5.4 Сводные метрики (верхняя панель)
- Всего объектов в портфеле
- Количество: в графике / риск отставания / отставание / завершено
- Средняя готовность по портфелю, %
- Среднее освоение бюджета, %

### 5.5 Фильтрация и поиск
- Фильтр по региону (выпадающий список)
- Фильтр по статусу
- Текстовый поиск по названию и региону
- Фильтры применяются одинаково в обоих режимах отображения

### 5.6 Хранение данных
- Данные сохраняются в персистентное хранилище, привязанное к пользователю (не публичное)
- При первом открытии — предзаполнение примерами/реальными проектами (seed-данные)
- Любое последующее изменение схемы данных сопровождается версионированием ключа хранения, чтобы не конфликтовать со старыми сохранёнными записями

## 6. Нефункциональные требования

| Категория | Требование |
|---|---|
| Платформа | Веб, открывается в браузере, без установки |
| Производительность | Мгновенный отклик при работе с портфелем до ~50 проектов |
| Доступность | Не требует подключения к интернету после загрузки (кроме сохранения данных) |
| Локализация | Интерфейс на русском языке |
| Визуальный стиль | Тематика диспетчерской/пункта управления ж/д (тёмная палитра, светофорные индикаторы, моноширинный шрифт для цифр) |
| Конфиденциальность | Данные не публичны, видны только пользователю, который их ввёл |

## 7. Метрики успеха

- Руководитель может назвать статус любого проекта из портфеля менее чем за 30 секунд после открытия приложения
- 100% активных проектов имеют заполненные поля риска и мероприятий, если статус ≠ «в графике»
- Время на обновление данных по одному проекту куратором — менее 2 минут (через табличный режим)

## 8. Открытые вопросы / риски продукта

1. **Источник истины.** Сейчас все данные вводятся вручную на основе презентаций подрядчиков — нет автоматической сверки с фактом. Риск рассинхронизации, если куратор забудет обновить данные.
2. **Множественность подрядчиков внутри проекта.** Текущая модель данных не отражает разбивку по участкам/субподрядчикам (как в презентациях по Бахты–Аягоз — 25 участков, 18 подрядчиков). При необходимости детализации потребуется расширение модели данных (вложенные участки).
3. **Бюджетные показатели.** Разные проекты по-разному определяют «% освоения» (от аванса, от факта СМР, от полной суммы договора) — требуется единая методология для сопоставимости показателей по портфелю.
4. **Совместное редактирование.** В v1 нет блокировок и истории изменений — при нескольких кураторах возможен конфликт «последний сохранивший побеждает».

## 10. Приложение: текущая реализация (справочно)

Версия v1 реализована как единый HTML-файл (`ktz_dashboard.html`), без серверной части, с использованием встроенного хранилища ключ-значение для персистентности данных пользователя. На момент написания документа в приложение внесены данные по 4 реальным проектам КТЖ (на основе презентаций по состоянию на 26–27.06.2026):

- Дарбаза – Мактаарал (1 очередь)
- Мойынты – Кызылжар
- Бахты – Аягоз (1 очередь)
- Обводная линия Алматы (Жетыген – Казыбек бек)
